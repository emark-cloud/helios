# Helios — Threat model

This document is a standalone, forensic-grade threat model for the Helios protocol intended for external auditors. It restates and expands the threats enumerated in `Helios.md §15.2` and binds each row to a concrete on-chain mitigation, modifier, or test in this repo. Authoritative source-of-truth remains `Helios.md §15` (trust assumptions, threats, explicit non-claims); when this file and the spec disagree, the spec wins. Last reviewed: **2026-05-08** 

---

## 1. Trust assumptions

The actors and systems Helios v1 trusts to varying degrees, sourced from `Helios.md §15.1` and cross-referenced against deployed addresses in `CLAUDE.md`:

- **End user (Passport identity).** Trust delegated to Particle Network MPC + Kite Passport (`@gokite-network/auth`, `gokite-aa-sdk`, Kite EntryPoint, SmartAccountFactory). Compromised passkey + email = AA wallet recovery. v2 considers a self-custody migration once `gokite-aa-sdk` exposes "claim ownership to external EOA".
- **Allocator operator.** *Limited trust.* On-chain bounds in `contracts/src/AllocatorVault.sol` enforce the user's meta-strategy; the operator can rank suboptimally but cannot exceed `maxCapital`, `maxPerStrategyBps`, or allocate to a disallowed class. Multiple competing allocators registered in `AllocatorRegistry`.
- **Strategy operator.** *Limited trust.* Cannot violate class invariants (Groth16 proof of class compliance is mandatory in `StrategyVault.executeWithProof` / `executeYieldRotationWithProof`). Can run a strategy that loses money — handled by reputation and the auto-defund path.
- **Helios price oracle signer (`ORACLE_SIGNER_PK`).** *Trusted.* Signs EIP-712 commits into `OraclePriceAnchor`/`OracleYieldAnchor`. The committed `oracle_root` is bound into every ZK proof and gates `executeWithProof`. Documented v1 centralization point — v2 path is Pyth/Chainlink adapter.
- **Helios reputation signer (`REPUTATION_SIGNER_PK`).** *Trusted.* Single signer in v1 posts to `ReputationAnchor`; v2 is multi-sig; v3 is ZK-attested computation. Engine code and inputs are public so any inflation is independently re-runnable.
- **Yield-market allowlist owner (Helios multi-sig).** Owner-only `StrategyRegistry.setMarketAllowlistRoot` curates lending venues for `yield_rotation_v1`.
- **NAV-divergence threshold owner (Helios multi-sig).** Owner-set `NAV_DIVERGENCE_THRESHOLD_BPS` (default 500 bps); slash itself is `onlyRegistry` → multi-sig.
- **LayerZero DVN set + relayers.** Default DVN configuration trusted for cross-chain reputation/attestation messages via `HeliosOApp`.
- **Trusted setup (Groth16 PTAU).** v1 uses local PTAU; production needs a real ceremony.
- **Hackathon judges.** Read-only consumers of the live frontend, subgraph, and `/judge` route — no on-chain trust, but assumed honest in their evaluation.

## 2. Threat table

Each row is one threat from `Helios.md §15.2`, rendered with vector and the specific on-chain or off-chain mitigation. Test paths are absolute repo-relative (`contracts/test/<File>.t.sol::<test>`). Line refs use `<file>:<line>` from the live code.

## 3. Residual risks (accepted in v1)

These risks remain open by design and are enumerated honestly per `Helios.md §15.3`:

- **Single-signer reputation engine.** Judges and users trust `REPUTATION_SIGNER_PK` not to inflate scores. Mitigated by public engine code + public on-chain inputs (anyone can re-run the math). v2 multi-sig + v3 ZK roadmap in `docs/reputation-math.md` and `docs/reputation-v1-v2-cutover.md`.
- **Single-signer price oracle, 180 s freshness window.** A live-compromised oracle key can mint a fraudulent root that validates trades for up to 180 s before the multi-sig revokes via `revokeRoot`. The yield anchor uses a separated typehash so price↔yield replay is impossible (`test_priceSig_cannotBeReplayedAsYield`). v2 replaces with Pyth/Chainlink adapter.
- **Helios multi-sig as ultimate authority.** Owner of `StrategyRegistry.setMarketAllowlistRoot`, `StrategyVault.setNavDivergenceThreshold`, `OraclePriceAnchor.setSigner / revokeRoot / unrevokeRoot`, `AllocatorRegistry.slash`, `StrategyRegistry.slash`, all UUPS `_authorizeUpgrade` calls, and pause/unpause across `UserVault`, `AllocatorVault`, `ReputationAnchor`. Compromise of the multi-sig is catastrophic. Out-of-scope mitigation: hardware-keyed multi-sig with documented signer custody.
- **LayerZero DVN trust.** Default DVN configuration is trusted. A malicious or compromised DVN majority could deliver a forged cross-chain message; the per-route nonce guard prevents replay but not first-time forgery from a compromised DVN.
- **Particle Network MPC trust (user-side).** Documented inheritance of Particle's MPC and Kite Passport security models — not Helios-mitigable.
- **Trusted-setup ceremony.** v1 uses local PTAU. A compromised setup invalidates Groth16 soundness. Production ceremony is post-hackathon (`Helios.md §16`).
- **Operator NAV under-protection (upper bound).** Without an on-chain per-asset price source, `reportNAV` cannot be checked for *over*-reporting on-chain in v1. Mitigated off-chain by the reputation engine and by realisation events. v2 adds bidirectional `markedNAV` (Helios.md §17 Phase 1).
- **Slashing dispute mechanism.** `slash` is owner-controlled; no on-chain dispute path in v1 (`Helios.md §16`).
- **Insurance fund.** No backstop for stake-exhaustion edge cases in v1 (`Helios.md §16`).
- **Strategy class versioning.** No on-chain migration path — a v2 momentum circuit would require a new class id and a fresh `StrategyVault` deployment.

## 4. Operational hardening pointers

Auditors and operators wanting to reproduce or harden the running stack should consult:

- **VPS / reverse-proxy hardening:** `deploy/README.md` documents the public surface (Nginx terminates 80/443; everything else loopback only) and the per-route rate limits in `deploy/nginx/helios.conf`. Phase 6 values (TODO.md line 475): **`helios_read` 100 r/min** for GET/HEAD/OPTIONS (burst 20), **`helios_write` 10 r/min** for POST/PUT/PATCH/DELETE (burst 2–3), **`helios_prover` 5 r/min** for Groth16 generation (burst 1). Method-based splitting via `$request_method` map; tripping a limit returns HTTP 429. Health endpoints are intentionally un-rate-limited. PM2 process supervision is described in `deploy/ecosystem.config.cjs`.
- **Signer-key custody.**
  - `ORACLE_SIGNER_PK` (price + yield anchors) and `REPUTATION_SIGNER_PK` are documented in `CLAUDE.md` "Environment variables" — kept out of the repo, sourced from VPS env or Vercel env, never committed.
  - Helios multi-sig holds owner role on every immutable contract listed in §3 above.
  - Kite Passport-backed wallets (`KITE_PASSPORT_SESSION_ID`) do not expose raw private keys (MPC-backed) — same variable name on testnet and mainnet, chain target switched via `KITE_PASSPORT_NETWORK`.
- **Phase-3 review HIGH-findings closure log:** `docs/phase-3-review.md` enumerates every HIGH issue and the redeploy that closed it; current address book in `CLAUDE.md` "Key addresses".
- **Phase-5 cross-chain review:** `docs/phase-5-review.md` covers the C1/C2/H3/H4/H5/H6 cross-chain follow-ups closed in PR #90; `docs/phase5-acceptance.md` is the WS8 evidence bundle (preflight script, multi-chain e2e, allocator-decision test).
- **Reputation v1→v2 cutover:** `docs/reputation-v1-v2-cutover.md` is the runbook for the Phase-5 redeploy that swaps the registry-bound anchor; today both `ReputationAnchor` (V1, registry-bound, `0x51c07adf…`) and `ReputationAnchorV2` (sidecar, `0x735680a3…`) are live.
- **Static analysis:** Slither + Mythril run in CI from Phase 6 polish (gating tag `v0.5.0`). Echidna property suites are scheduled alongside the Phase-1 external audit and are explicitly out-of-scope for v1 (`Helios.md §16`).
- **Audit checklist:** `docs/audit-checklist.md` is a planned post-hackathon deliverable (not yet shipped at the time of this review). Until it lands, this document plus `docs/phase-3-review.md` and `docs/phase-5-review.md` are the canonical pre-audit references.
- **Reproduce path:** `scripts/e2e-scenario.sh` (default) and `scripts/e2e-scenario.sh phase5` (cross-chain) run against a fresh stack. Phase-5 preflight in `scripts/preflight-phase5.sh` and latency profiling in `scripts/measure_xchain_latency.py`.
